## 3.1 Alerting Philosophy & Best Practices

Alert fatigue is one of the most damaging problems in production operations. Too many alerts → engineers stop reading them → critical issues get missed.

**The golden rule: ALERT ON SYMPTOMS, NOT CAUSES.**

| ✅ Alert on symptoms (user-visible, worth waking someone up) | ❌ Don't alert on causes |
|---|---|
| Patients cannot book appointments (HTTP 5xx rate high) | CPU usage is 85% (might be fine) |
| Appointment service responding slowly (p99 > 2s) | JVM GC pause exceeded 200ms (one pause) |
| Notification emails not being sent (failure rate > 10%) | DB connection pool at 80% capacity |
| Billing service is down (`up == 0`) | |

### Rules for Good Alerts

1. **Every alert must be ACTIONABLE** — someone receives it, looks at the runbook, and does something. If there is no action, it is noise.
2. **Every alert must have a RUNBOOK** — a document that says: "When you see this alert, here's what to check and how to fix it."
3. **Prefer MULTI-WINDOW alerting** — use the `for` clause to require a condition to hold before firing. An anomaly for 30 seconds is not an incident.
4. **Tune thresholds from REAL DATA** — look at your 30-day p99 baseline and set your alert at 3× the baseline. Revisit quarterly.
5. **Avoid FLAPPING alerts** — use appropriate `for` durations and stable thresholds.

### ClinicCare Alert Tiers

| Tier | Condition Examples | Response |
|---|---|---|
| **Page** (urgent, off-hours) | Service down, error rate > 5%, p99 > 3s | Immediate PagerDuty / on-call |
| **Ticket** (business hours) | Error rate rising, disk > 80%, pool nearing limit | Next business day |
| **Informational** | New deployment, config change, scale event | Log only |

## 3.2 SLI, SLO, SLA Basics

### SLI — Service Level Indicator

A specific, measurable metric that reflects the quality of a service.

**ClinicCare SLI examples:**
- Availability SLI: percentage of appointment booking requests that succeed (HTTP 2xx / all requests)
- Latency SLI: percentage of booking requests completing in < 500ms
- Error rate SLI: rate of 5xx errors from `billing-service`

### SLO — Service Level Objective

The **internal** target for an SLI. This is your internal commitment to quality.

**ClinicCare SLO examples:**
- Availability SLO: 99.9% of appointment booking requests succeed per month (~43 minutes of allowed downtime)
- Latency SLO: 95% of booking requests complete in < 500ms
- Notification SLO: 99% of notifications sent within 30 seconds

### SLA — Service Level Agreement

The **external, contractual** commitment made to customers. Violating an SLA often has financial consequences (credits, penalties). SLA is typically less aggressive than SLO.

> If SLO = 99.9%, SLA might be 99.5% (buffer for negotiation and error budget).

### Error Budget

```
Error budget = 1 - SLO

If SLO is 99.9% availability per month:
Error budget = 0.1% of 43,200 minutes ≈ 43 minutes of allowed downtime
```

When you burn through your error budget, you stop deploying new features until you restore reliability. This creates a natural balance between velocity (new features) and reliability (stability).

### PromQL for SLO Tracking

```promql
# Current availability ratio (last 30 days)
1 - (
  sum(increase(http_server_requests_seconds_count{job="appointment-service", outcome="SERVER_ERROR"}[30d]))
  /
  sum(increase(http_server_requests_seconds_count{job="appointment-service"}[30d]))
)

# Latency SLO compliance: % of requests under 500ms
sum(rate(http_server_requests_seconds_bucket{job="appointment-service", le="0.5"}[5m]))
/
sum(rate(http_server_requests_seconds_count{job="appointment-service"}[5m]))
```

## 3.3 Writing Alert Rules

Alert rules are written in YAML files and loaded via `prometheus.yml` `rule_files`.

### Basic Syntax

```yaml
groups:
  - name: group_name
    rules:
      - alert: AlertName
        expr: <PromQL expression>
        for: <duration>         # How long condition must be true before firing
        labels:
          severity: <critical|warning|info>
          team:     <team-name>
        annotations:
          summary:     "<short description — shown in notification title>"
          description: "<detailed description — shown in notification body>"
          runbook_url: "<link to runbook>"
```

### The `for` Clause

| Value | Meaning | Use case |
|---|---|---|
| `for: 0m` | Fire immediately when true | Use with caution (very noisy) |
| `for: 2m` | Fire only if condition holds 2 min | Short-lived spikes that matter |
| `for: 5m` | Standard for most production alerts | Service errors, latency |
| `for: 15m` | Slow-burn conditions | Disk filling, gradual memory leak |

### Alert States

```
INACTIVE  →  Expression is false. No alert.
PENDING   →  Expression true, but 'for' duration not yet elapsed.
FIRING    →  Condition held for 'for' duration. Alert dispatched to Alertmanager.
RESOLVED  →  Condition became false again. Alertmanager sends resolution notice.
```

## 3.4 ClinicCare Alert Rules — Full Production Set

```yaml
# ~/cliniccare-monitoring/prometheus/rules/cliniccare_alerts.yml

groups:

  # === SERVICE AVAILABILITY ===
  - name: cliniccare_availability
    rules:

      - alert: ClinicCareServiceDown
        expr: up{job=~".*-service"} == 0
        for: 1m
        labels:
          severity: critical
          team:     platform
        annotations:
          summary:     "ClinicCare service is down: {{ $labels.job }}"
          description: >
            Service {{ $labels.job }} on {{ $labels.instance }} has been
            unreachable for more than 1 minute.
          runbook_url: "https://wiki.cliniccare.internal/runbooks/service-down"

      - alert: ClinicCareNodeExporterDown
        expr: up{job="node-exporter"} == 0
        for: 2m
        labels:
          severity: warning
          team:     platform
        annotations:
          summary: "Node Exporter is down — host metrics unavailable"

  # === HTTP ERROR RATES ===
  - name: cliniccare_error_rates
    rules:

      - alert: HighErrorRateCritical
        expr: >
          (
            sum by (job) (rate(http_server_requests_seconds_count{outcome="SERVER_ERROR"}[5m]))
            /
            sum by (job) (rate(http_server_requests_seconds_count[5m]))
          ) > 0.05
        for: 5m
        labels:
          severity: critical
          team:     backend
        annotations:
          summary: "High HTTP error rate on {{ $labels.job }}"
          description: >
            {{ $labels.job }} is returning {{ printf "%.1f" (mul $value 100) }}%
            HTTP 5xx errors (threshold: 5%). This is impacting patients and staff.
          runbook_url: "https://wiki.cliniccare.internal/runbooks/high-error-rate"

      - alert: HighErrorRateWarning
        expr: >
          (
            sum by (job) (rate(http_server_requests_seconds_count{outcome="SERVER_ERROR"}[5m]))
            /
            sum by (job) (rate(http_server_requests_seconds_count[5m]))
          ) > 0.01
        for: 10m
        labels:
          severity: warning
          team:     backend
        annotations:
          summary: "Elevated error rate on {{ $labels.job }}"
          description: >
            {{ $labels.job }} has a {{ printf "%.1f" (mul $value 100) }}% error rate
            (threshold: 1%). Monitor closely.

  # === LATENCY ===
  - name: cliniccare_latency
    rules:

      - alert: AppointmentBookingSlowCritical
        expr: >
          histogram_quantile(0.99,
            sum by (le) (
              rate(appointment_booking_duration_seconds_bucket{job="appointment-service"}[5m])
            )
          ) > 3.0
        for: 5m
        labels:
          severity: critical
          team:     backend
        annotations:
          summary: "Appointment booking is critically slow (p99 > 3s)"
          description: >
            p99 booking time is {{ $value | humanizeDuration }}. Patients are
            experiencing significant delays.

      - alert: HTTPLatencyHighWarning
        expr: >
          histogram_quantile(0.95,
            sum by (job, le) (rate(http_server_requests_seconds_bucket[5m]))
          ) > 1.0
        for: 10m
        labels:
          severity: warning
          team:     backend
        annotations:
          summary: "High p95 latency on {{ $labels.job }}"
          description: "p95 response time is {{ $value | humanizeDuration }}. SLO threshold: 500ms."

  # === BUSINESS METRICS ===
  - name: cliniccare_business
    rules:

      - alert: NotificationFailureRateHigh
        expr: >
          (
            sum by (channel) (rate(notification_failures_total[5m]))
            /
            sum by (channel) (rate(notifications_sent_total[5m]))
          ) > 0.10
        for: 5m
        labels:
          severity: warning
          team:     notifications
        annotations:
          summary: "High notification failure rate on {{ $labels.channel }}"
          description: >
            {{ printf "%.0f" (mul $value 100) }}% of {{ $labels.channel }}
            notifications are failing. Patients may not receive confirmations.

      - alert: AppointmentQueueDepthHigh
        expr: appointment_queue_depth > 50
        for: 10m
        labels:
          severity: warning
          team:     backend
        annotations:
          summary: "Appointment queue depth is high: {{ $value }} pending"

      - alert: NoBillingInvoicesGenerated
        expr: increase(billing_invoices_generated_total[1h]) == 0
        for: 0m
        labels:
          severity: warning
          team:     billing
        annotations:
          summary: "No invoices generated in the last hour"

  # === INFRASTRUCTURE ===
  - name: cliniccare_infrastructure
    rules:

      - alert: HighCPUUsage
        expr: >
          100 - (avg by (instance) (
            rate(node_cpu_seconds_total{mode="idle"}[5m])
          ) * 100) > 85
        for: 15m
        labels:
          severity: warning
          team:     platform
        annotations:
          summary: "High CPU usage on {{ $labels.instance }}"

      - alert: HighMemoryUsage
        expr: >
          (1 - (node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes)) * 100 > 90
        for: 5m
        labels:
          severity: critical
          team:     platform
        annotations:
          summary: "Memory utilisation critical on {{ $labels.instance }}"
          description: "Memory is {{ printf \"%.0f\" $value }}%. Risk of OOM kill."

      - alert: DiskSpaceLow
        expr: >
          (1 - node_filesystem_avail_bytes{mountpoint="/"} /
               node_filesystem_size_bytes{mountpoint="/"}) * 100 > 80
        for: 5m
        labels:
          severity: warning
          team:     platform
        annotations:
          summary: "Low disk space on {{ $labels.instance }}"

      - alert: DiskWillFillIn4Hours
        expr: >
          predict_linear(node_filesystem_avail_bytes{mountpoint="/"}[1h], 4 * 3600) < 0
        for: 30m
        labels:
          severity: critical
          team:     platform
        annotations:
          summary: "Disk predicted to be full within 4 hours"

      - alert: JVMHeapCritical
        expr: >
          jvm_memory_used_bytes{area="heap"} /
          jvm_memory_max_bytes{area="heap"} > 0.90
        for: 5m
        labels:
          severity: critical
          team:     backend
        annotations:
          summary: "JVM heap critical on {{ $labels.job }}"
          description: "JVM heap utilisation above 90%. Risk of OutOfMemoryError."

      - alert: DBConnectionPoolExhausted
        expr: hikaricp_connections_pending > 0
        for: 2m
        labels:
          severity: warning
          team:     backend
        annotations:
          summary: "DB connection pool has waiting threads on {{ $labels.job }}"
          description: "{{ $value }} threads waiting for a DB connection. Will cause request timeouts."
```

## 3.5 Alertmanager Architecture

Alertmanager receives alerts from Prometheus and:

1. **DEDUPLICATES** — groups identical alerts together
2. **GROUPS** — batches related alerts to avoid notification storms
3. **ROUTES** — sends alerts to the right receiver based on labels
4. **INHIBITS** — suppresses certain alerts when others are firing
5. **SILENCES** — allows temporary muting during maintenance

```
[Prometheus Alert Rule fires]
        ↓
[Alertmanager receives alert via HTTP POST]
        ↓
[Routing tree: which receiver?]
        ↓
[Grouping: batch related alerts]
        ↓
[Wait for group_wait (30s) — more alerts may arrive for same group]
        ↓
[Send to receiver: Email / Slack / PagerDuty]
        ↓
[Wait group_interval (5m) — send updates for ongoing group]
        ↓
[Alert resolves → send resolution notification]
```

## 3.6 Alertmanager Configuration

```yaml
# ~/cliniccare-monitoring/alertmanager/alertmanager.yml

global:
  smtp_smarthost:      'smtp.gmail.com:587'
  smtp_from:           'cliniccare-alerts@example.com'
  smtp_auth_username:  'cliniccare-alerts@example.com'
  smtp_auth_password:  'your-app-password'
  smtp_require_tls:    true
  resolve_timeout:     5m

templates:
  - '/etc/alertmanager/templates/*.tmpl'

route:
  receiver:       'default-slack'
  group_by:       ['alertname', 'job']
  group_wait:     30s
  group_interval: 5m
  repeat_interval: 4h

  routes:
    # Critical → PagerDuty (fast)
    - match:
        severity: critical
      receiver: 'pagerduty-critical'
      group_wait:      10s
      repeat_interval: 1h

    # Team-based routing
    - match: { team: billing }
      receiver: 'slack-billing'

    - match: { team: platform }
      receiver: 'slack-platform'

    - match: { team: backend }
      receiver: 'slack-backend'

inhibit_rules:
  # If service is completely down, suppress warning-level alerts for same job
  - source_match:
      alertname: 'ClinicCareServiceDown'
      severity:  'critical'
    target_match:
      severity:  'warning'
    equal: ['job']

  # If high memory (critical), suppress JVM heap critical (same root cause)
  - source_match:
      alertname: 'HighMemoryUsage'
    target_match:
      alertname: 'JVMHeapCritical'
    equal: ['instance']

receivers:
  - name: 'default-slack'
    slack_configs:
      - api_url:      'https://hooks.slack.com/services/YOUR/WEBHOOK/URL'
        channel:      '#cliniccare-alerts'
        send_resolved: true
        color: '{{ if eq .Status "firing" }}danger{{ else }}good{{ end }}'

  - name: 'slack-backend'
    slack_configs:
      - api_url: 'https://hooks.slack.com/services/YOUR/WEBHOOK/URL'
        channel: '#cliniccare-backend-alerts'
        send_resolved: true

  - name: 'slack-billing'
    slack_configs:
      - api_url: 'https://hooks.slack.com/services/YOUR/WEBHOOK/URL'
        channel: '#cliniccare-billing-alerts'
        send_resolved: true

  - name: 'slack-platform'
    slack_configs:
      - api_url: 'https://hooks.slack.com/services/YOUR/WEBHOOK/URL'
        channel: '#cliniccare-platform-alerts'
        send_resolved: true

  - name: 'email-oncall'
    email_configs:
      - to:           'oncall-engineer@cliniccare.com'
        send_resolved: true
        headers:
          Subject: '[{{ .Status | toUpper }}] {{ .CommonAnnotations.summary }}'

  - name: 'pagerduty-critical'
    pagerduty_configs:
      - routing_key:  'YOUR_PAGERDUTY_ROUTING_KEY'
        description:  '{{ .CommonAnnotations.summary }}'
```

## 3.7 Routing, Grouping, Inhibition, Silencing

### Routing

The routing tree is evaluated top-down; the first matching route wins. Use `continue: true` to send an alert to multiple receivers:

```yaml
routes:
  - match:
      severity: critical
    receiver: 'pagerduty-critical'
    continue: true         # Keep evaluating — also send to Slack
  - match:
      severity: critical
    receiver: 'slack-oncall'
```

### Grouping Settings

| Setting | Meaning |
|---|---|
| `group_by: ['alertname', 'job']` | Alerts with the same `alertname` + `job` fire as ONE notification |
| `group_wait: 30s` | After first alert in a group, wait 30s for more before sending |
| `group_interval: 5m` | After sending, wait 5m before sending again for that group |
| `repeat_interval: 4h` | Re-notify if alert group is still firing after 4h |

### Inhibition

Suppresses child alerts when a parent alert is firing.

```yaml
inhibit_rules:
  - source_match:
      alertname: 'ClinicCareServiceDown'
    target_match:
      severity: 'warning'
    equal: ['job']
```

> Without inhibition: if `appointment-service` goes down, you get 5 separate alerts (down, high error rate, high latency, missing metrics…). With inhibition: just one — *service is down*.

### Silences

Temporary muting during planned maintenance:

```bash
# Create a 2-hour silence via API
curl -X POST http://localhost:9093/api/v2/silences \
  -H "Content-Type: application/json" \
  -d '{
    "matchers": [
      {"name": "job",      "value": "billing-service", "isRegex": false},
      {"name": "severity", "value": "warning",          "isRegex": false}
    ],
    "startsAt":  "2026-03-10T22:00:00Z",
    "endsAt":    "2026-03-11T00:00:00Z",
    "createdBy": "vaman.deshmukh",
    "comment":   "Planned maintenance: billing-service database migration"
  }'
```

## 3.8 Starting Alertmanager

```bash
# Step 1 — Create directory
mkdir -p ~/cliniccare-monitoring/alertmanager

# Step 2 — Create alertmanager.yml (see section 3.6)

# Step 3 — Start Alertmanager
docker run -d \
  --name alertmanager \
  --network host \
  -v ~/cliniccare-monitoring/alertmanager/alertmanager.yml:/etc/alertmanager/alertmanager.yml \
  -p 9093:9093 \
  prom/alertmanager:latest \
  --config.file=/etc/alertmanager/alertmanager.yml

# Step 4 — Update prometheus.yml alerting section
# alerting:
#   alertmanagers:
#     - static_configs:
#         - targets: ['localhost:9093']

# Step 5 — Reload Prometheus
curl -X POST http://localhost:9090/-/reload
```

Verify: **Prometheus UI → Status → Runtime & Build Info** — Alertmanager should be listed as connected.

## Day 3 Summary & Key Takeaways

| # | Takeaway |
|---|---|
| ✓ | Alert on symptoms (user-visible impact), not causes (internal metrics). Every alert must be actionable. |
| ✓ | The `for` clause prevents alert flapping. Choose durations based on urgency. |
| ✓ | Alert states: `INACTIVE → PENDING → FIRING → RESOLVED`. PENDING means condition is true but `for` not elapsed yet. |
| ✓ | SLIs measure what matters; SLOs define the target; error budget determines deployment risk. |
| ✓ | Alertmanager handles routing, grouping, inhibition, and silencing. Group by `alertname` + `job` to prevent storms. |
| ✓ | Inhibition rules suppress child alerts when a parent (service down) is firing. |
| ✓ | Use silences for planned maintenance — never as a way to ignore real alerts. |
