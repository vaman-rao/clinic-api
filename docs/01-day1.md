## 1.1 Monitoring vs Observability

**Monitoring** is the practice of collecting predefined metrics and checking them against known thresholds. It answers: *"Is the thing I am watching broken?"*

**Observability** is the property of a system that lets you understand its internal state from its external outputs — without having to deploy new instrumentation every time a new question arises. It answers: *"Why is the thing broken, and what is happening inside?"*

| | Monitoring | Observability |
|---|---|---|
| Answers | **THAT** something is wrong | **WHY and WHERE** it is wrong |
| Approach | Pre-defined checks / thresholds | Exploratory; ask new questions without redeploy |
| Tooling | Nagios, Zabbix, CloudWatch Alarms | Prometheus + Grafana + Jaeger + Loki |

**ClinicCare example:**
- *Monitoring:* "Appointment service response time > 2 seconds. Alert fired."
- *Observability:* "Response time spiked because the DB connection pool in `appointment-service` was exhausted when bulk-import of 3,000 records ran at 14:00. We can see this from metrics, correlated with slow-query logs and a trace showing the specific DB call that stalled."

**Why observability matters in microservices:**
- Requests span multiple services: `patient → appointment → notification`
- Any hop can introduce latency or errors
- Without observability you are blind to cascading failures

## 1.2 The Observability Triad

Modern observability rests on three pillars:

| Pillar | What it is | ClinicCare Example |
|---|---|---|
| **Metrics** | Numeric measurements over time. Efficient, aggregatable. | `appointment_service_requests_total{status="200"}` |
| **Logs** | Timestamped text records of events. Verbose, searchable. | `ERROR PatientService - DB timeout after 30s` |
| **Traces** | End-to-end journey of a single request across services. | Patient registration → appointment creation → notification dispatch |

> This bootcamp focuses on **Metrics** (Prometheus) and **Visualisation** (Grafana).  
> Tracing (Jaeger / Tempo) and log aggregation (Loki) are adjacent tools you will encounter in enterprise stacks.

**Prometheus's role in the triad:**
- Collects and stores metrics
- Provides a query language (PromQL) to analyse them
- Triggers alerts when thresholds are crossed
- Integrates with Grafana for dashboards

## 1.3 Prometheus Architecture

Prometheus follows a **PULL model** — it actively scrapes (fetches) metrics from target endpoints at a configurable interval, rather than waiting for targets to push data.

```
[Targets (apps/exporters)] <── HTTP GET /metrics ── [Prometheus Server]
                                                             │
                                              ┌──────────────┴──────────────┐
                                              │      Time-Series DB          │
                                              │       (local disk)           │
                                              └──────────────┬──────────────┘
                                                             │
                                         ┌───────────────────┼──────────────┐
                                         │                   │              │
                                      [PromQL]        [Alertmanager]    [Grafana]
                                      (queries)           (alerts)     (dashboards)
```

**Key components:**

| Component | Role |
|---|---|
| **Prometheus Server** | Core engine. Scrapes, stores, evaluates rules, exposes HTTP API and UI. |
| **Targets** | Any process exposing a `/metrics` endpoint. In our case: each of the 5 ClinicCare services + Node Exporter. |
| **Exporters** | Adapters that expose metrics for third-party systems (databases, OS, hardware) that cannot instrument themselves. |
| **Alertmanager** | Receives alerts from Prometheus, deduplicates, groups, routes, and dispatches them (email, Slack, PagerDuty). |
| **Pushgateway** | For short-lived jobs (batch, cron) that cannot be scraped. They push metrics to the Pushgateway which Prometheus then scrapes. *(Not used in this bootcamp — mentioned for completeness.)* |
| **Service Discovery** | Mechanism to automatically discover scrape targets in dynamic environments (Kubernetes, Consul, EC2). Covered in Day 2. |

## 1.4 Time-Series & Labels

A **time-series** is a sequence of `(timestamp, value)` pairs identified by a unique metric name plus a set of `key=value` labels.

```
metric_name{label1="value1", label2="value2", ...}
```

**Examples from ClinicCare:**
```
http_requests_total{service="appointment-service", method="POST", status="200"}
http_requests_total{service="appointment-service", method="POST", status="500"}
http_requests_total{service="patient-service",     method="GET",  status="200"}
```

Each unique combination of metric name + label set is a **separate time-series**.

### Label Best Practices

| ✅ Good label candidates | ❌ Avoid high-cardinality labels |
|---|---|
| `service`, `endpoint`, `method` | `user_id` (millions of values) |
| `status_code`, `region`, `env` | `request_id`, IP address |

> Every unique label combination creates a new time-series (cardinality). High cardinality → millions of time-series → memory and performance problems.

## 1.5 Metric Types

Prometheus has four native metric types:

### 1. Counter
- Always increases (or resets to zero on restart)
- Counts occurrences of events
- **Never** use the raw value — always use `rate()` or `increase()`

**ClinicCare counters:**
```
appointments_booked_total        # total appointments ever created
patient_registrations_total      # total patients ever registered
billing_invoices_generated_total # total invoices generated
http_server_requests_seconds_count
```

### 2. Gauge
- Can go up or down freely
- Represents current state / value at a point in time

**ClinicCare gauges:**
```
active_doctors_online            # doctors currently logged in
appointment_queue_depth          # appointments waiting to be processed
db_connection_pool_active        # connections in use right now
jvm_memory_used_bytes            # current JVM heap usage
```

### 3. Histogram
- Samples observations and counts them in configurable buckets
- Automatically creates three time-series:
  - `_bucket{le="..."}` — cumulative count per bucket boundary
  - `_sum` — sum of all observed values
  - `_count` — total number of observations
- Used for: request durations, response sizes, queue wait times

**ClinicCare histogram example:**
```
appointment_booking_duration_seconds_bucket{le="0.1"}
appointment_booking_duration_seconds_bucket{le="0.5"}
appointment_booking_duration_seconds_bucket{le="1.0"}
appointment_booking_duration_seconds_sum
appointment_booking_duration_seconds_count
```

### 4. Summary
- Similar to histogram but calculates quantiles on the **client side**
- Less flexible for aggregation across instances
- **Prefer histograms in production** (they support server-side aggregation)

**ClinicCare summary example:**
```
notification_dispatch_duration_seconds{quantile="0.95"}
```

### Decision Guide

| Use case | Metric type |
|---|---|
| Counting events (requests, errors, logins) | **Counter** |
| Current value / state | **Gauge** |
| Measuring durations or sizes | **Histogram** (preferred) |
| Client-side quantiles, fixed service instances | **Summary** |

## 1.6 Installing Prometheus (Docker-Based)

> In Day 4 we use Helm / kube-prometheus-stack for Kubernetes. Today we start simple with Docker.

### Step 1 — Create directory structure

```bash
mkdir -p ~/cliniccare-monitoring/prometheus
cd ~/cliniccare-monitoring/prometheus
```

### Step 2 — Create `prometheus.yml`

```yaml
# prometheus.yml
global:
  scrape_interval:     15s   # How often to scrape targets
  evaluation_interval: 15s   # How often to evaluate alert rules
  scrape_timeout:      10s   # Timeout for each scrape

alerting:
  alertmanagers:
    - static_configs:
        - targets: []         # Fill in on Day 3

rule_files: []                # Alert rule files — added on Day 3

scrape_configs:
  # Prometheus scraping itself
  - job_name: 'prometheus'
    static_configs:
      - targets: ['localhost:9090']

  # Node Exporter (host OS metrics)
  - job_name: 'node-exporter'
    static_configs:
      - targets: ['node-exporter:9100']
```

### Step 3 — Start Node Exporter

```bash
docker run -d \
  --name node-exporter \
  --network host \
  prom/node-exporter:latest
```

### Step 4 — Start Prometheus

```bash
docker run -d \
  --name prometheus \
  --network host \
  -v ~/cliniccare-monitoring/prometheus/prometheus.yml:/etc/prometheus/prometheus.yml \
  -p 9090:9090 \
  prom/prometheus:latest \
  --config.file=/etc/prometheus/prometheus.yml \
  --storage.tsdb.path=/prometheus \
  --web.enable-lifecycle
```

### Step 5 — Verify

Open `http://localhost:9090` → **Status → Targets**  
Both `prometheus` and `node-exporter` should show state **UP**.

## 1.7 `prometheus.yml` Deep Dive

### `global` section

```yaml
global:
  scrape_interval:     15s
  evaluation_interval: 15s
  external_labels:           # Added to all outgoing metrics (federation/remote)
    environment: 'dev'
    cluster:     'cliniccare-local'
```

| Setting | Typical Value | Notes |
|---|---|---|
| `scrape_interval` | 15s | Reduce to 5s for fast-changing alert metrics |
| `evaluation_interval` | 15s | Should match or be a multiple of `scrape_interval` |
| `scrape_timeout` | 10s | Must be ≤ `scrape_interval` |

### `scrape_configs` — full reference

```yaml
scrape_configs:
  - job_name: 'appointment-service'     # Logical name for this group
    metrics_path: '/actuator/prometheus' # Override default /metrics
    scheme: 'http'                       # http or https
    scrape_interval: 10s                 # Override global for this job
    scrape_timeout:  8s
    static_configs:
      - targets:
          - 'appointment-service:8082'   # host:port of target
        labels:
          env:  'development'
          team: 'clinic-backend'
```

> The `job_name` label is automatically added to every metric scraped. So a metric from `appointment-service` will always carry `job="appointment-service"` and `instance="appointment-service:8082"`.

## 1.8 Node Exporter

Node Exporter is a Prometheus exporter for hardware and OS metrics. It runs as a daemon and exposes metrics at `:9100/metrics`.

**Key metrics exposed:**

```
node_cpu_seconds_total{mode="idle|user|system|iowait|..."}
node_memory_MemAvailable_bytes
node_memory_MemTotal_bytes
node_filesystem_avail_bytes{mountpoint="/"}
node_filesystem_size_bytes{mountpoint="/"}
node_network_receive_bytes_total{device="eth0"}
node_network_transmit_bytes_total{device="eth0"}
node_load1 / node_load5 / node_load15    # system load averages
```

**In the ClinicCare context, these answer questions like:**
- Is the host running out of memory as `doctor-service` processes large queries?
- Is disk usage growing because `billing-service` generates large PDF invoices?
- Is network I/O spiking because `notification-service` sends bulk SMS?

## 1.9 Introduction to PromQL

PromQL (Prometheus Query Language) is a functional expression language for selecting and aggregating time-series data.

### Instant Vector Selector

Returns one sample per matching time-series at the current moment.

```promql
http_requests_total
http_requests_total{job="appointment-service"}
http_requests_total{job="appointment-service", status=~"5.."}
```

**Label matchers:**

| Operator | Meaning | Example |
|---|---|---|
| `=` | Exact match | `{service="patient-service"}` |
| `!=` | Not equal | `{status!="200"}` |
| `=~` | Regex match | `{status=~"4.."}` |
| `!~` | Negative regex | `{method!~"GET\|HEAD"}` |

### Range Vector Selector

Returns a range of samples (a window) for each time-series.

```promql
http_requests_total[5m]   # Last 5 minutes of samples
http_requests_total[1h]   # Last 1 hour
```

Duration units: `s` (seconds), `m` (minutes), `h` (hours), `d` (days), `w` (weeks)

### Essential Functions

```promql
# Rate — per-second average increase (use on counters, for dashboards)
rate(http_requests_total[5m])

# iRate — instant rate using last two points (use for alerting)
irate(http_requests_total[5m])

# Increase — total increase over interval
increase(appointments_booked_total[1h])
# "How many appointments were booked in the last hour?"
```

### Arithmetic & Comparison

```promql
# Memory utilisation %
(1 - (node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes)) * 100

# Only series where value exceeds threshold
http_requests_total > 1000
```

### Lab 1.3 — Write PromQL for CPU, Memory, Disk

**Query 1 — CPU Utilisation %:**
```promql
100 - (avg by (instance) (rate(node_cpu_seconds_total{mode="idle"}[5m])) * 100)
```

**Query 2 — Memory Utilisation %:**
```promql
(1 - (node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes)) * 100
```

**Query 3 — Disk Space Used %:**
```promql
(1 - node_filesystem_avail_bytes{mountpoint="/"} /
     node_filesystem_size_bytes{mountpoint="/"}) * 100
```

**Query 4 — Network Receive Rate (bytes/sec):**
```promql
rate(node_network_receive_bytes_total{device="eth0"}[5m])
```

**Challenge Query:**  
Write a query that shows which filesystem mount point has the least free space (absolute bytes).  
*Hint:* `min by (mountpoint) (node_filesystem_avail_bytes)`
