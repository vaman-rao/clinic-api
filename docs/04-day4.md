## 4.1 Grafana Architecture

Grafana is an open-source analytics and visualisation platform. It connects to data sources and renders dashboards, panels, and alerts.

### Core Concepts

| Concept | Description |
|---|---|
| **Data Sources** | Connections to backends where data lives (Prometheus, Loki, PostgreSQL…) |
| **Dashboards** | Collections of panels arranged in a grid |
| **Panels** | Individual visualisations (graph, stat, table, gauge, etc.) |
| **Variables** | Dynamic values that parameterise queries (service selector, environment) |
| **Playlists** | Auto-rotating dashboard views (for NOC / ops rooms) |
| **Folders** | Organise dashboards by team or domain |
| **Annotations** | Mark events (deployments, incidents) on graphs |
| **Unified Alerting** | Grafana's own alert engine (introduced in 8.x) |

**Grafana vs Prometheus UI:**
- **Prometheus UI:** for ad-hoc exploration and debugging PromQL
- **Grafana:** for persistent, shareable dashboards and production monitoring

### Starting Grafana via Docker

```bash
docker run -d \
  --name grafana \
  --network host \
  -p 3000:3000 \
  -v ~/cliniccare-monitoring/grafana:/var/lib/grafana \
  -e GF_SECURITY_ADMIN_PASSWORD=cliniccare123 \
  grafana/grafana:latest
```

Access: `http://localhost:3000` · Login: `admin / cliniccare123`

## 4.2 Connecting Prometheus as a Data Source

### Manual (UI)

1. **Connections → Data Sources → Add data source**
2. Select **Prometheus**
3. Configure:
   - Name: `ClinicCare Prometheus`
   - URL: `http://localhost:9090`
   - Access: `Server` (default)
4. Click **Save & Test** → "Data source connected."

### Provisioning (Infrastructure-as-Code — preferred)

```yaml
# ~/cliniccare-monitoring/grafana/provisioning/datasources/prometheus.yml
apiVersion: 1

datasources:
  - name:       ClinicCare-Prometheus
    type:       prometheus
    access:     proxy
    url:        http://localhost:9090
    isDefault:  true
    editable:   false
    jsonData:
      httpMethod:   POST
      manageAlerts: true
      timeInterval: 15s
```

Mount when starting Grafana:
```bash
-v ~/cliniccare-monitoring/grafana/provisioning:/etc/grafana/provisioning
```

Grafana auto-loads the data source on startup. No manual UI steps needed. This is the **GitOps-friendly** production pattern.

## 4.3 Panels, Dashboards, Variables & Templating

### Panel Types — Choosing the Right Visualisation

| Panel Type | Best For | Example |
|---|---|---|
| **Time Series** | Any metric that changes over time | Request rate, error rate, latency |
| **Stat** | Single current value | Active doctors online, error count last hour |
| **Gauge** | Utilisation between min/max | CPU %, memory %, error rate % |
| **Bar Gauge** | Comparing multiple values | Request rate per service side by side |
| **Table** | Tabular data | Top 10 slowest endpoints |
| **Pie Chart** | Proportional view | Request distribution by service |
| **Heatmap** | Histogram over time | Latency distributions (p50, p90, p99 over a day) |

### Building a Panel — Step by Step

1. In a dashboard → **Add → Visualization**
2. Select your data source (ClinicCare-Prometheus)
3. Enter PromQL expression in the query box
4. Select panel type from the right pane
5. Set **title**, **description**, **units** (bytes, seconds, %, req/s)
6. Add **thresholds** (turns the panel red/orange when crossed)
7. Click **Apply**

### Important Panel Settings

```
Unit: Always set the unit. Grafana auto-formats:
  'bytes(SI)'       → "1.2 MB" instead of "1248576"
  'seconds'         → "3.2s"  instead of "3.21"
  'percent (0-100)' → "85.3%"
  'reqps'           → "42.3 req/s"

Legend format: use {{label_name}} e.g. "{{job}} - {{method}}"

Thresholds: for error rate: 0 = green, 0.01 = yellow, 0.05 = red
```

### Variables & Templating

Variables make dashboards interactive and reusable — they appear as dropdowns at the top.

**Example — `$service` variable** (select which ClinicCare service to view):

In **Dashboard Settings → Variables → Add variable:**

```
Name        : service
Type        : Query
Data Source : ClinicCare-Prometheus
Query       : label_values(up{job=~".*-service"}, job)
Multi-value : Yes
Include All : Yes
```

Usage in panel queries:
```promql
rate(http_server_requests_seconds_count{job="$service"}[5m])
```

### Common Variable Types

| Type | Use Case |
|---|---|
| **Query** | Values from Prometheus (`label_values`, `query_result`) |
| **Custom** | Fixed list: `env: dev, staging, production` |
| **Interval** | Time grouping: `1m, 5m, 15m, 1h` |
| **Text** | Free text input (filter by pod name, user, etc.) |

**Built-in `$__rate_interval`:**
```promql
rate(http_requests_total[$__rate_interval])
```
Grafana auto-computes the appropriate interval based on the dashboard time range and step.

## 4.4 ClinicCare Dashboards — Design & Queries

### Dashboard 1: ClinicCare Infrastructure Overview

**Panel: CPU Utilisation %**
```promql
100 - (avg by (instance) (rate(node_cpu_seconds_total{mode="idle"}[5m])) * 100)
```
- Visualization: Time Series · Thresholds: 70=yellow, 85=red · Unit: Percent (0-100)

**Panel: Memory Utilisation %**
```promql
(1 - (node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes)) * 100
```
- Visualization: Gauge · Thresholds: 75=yellow, 90=red · Min: 0, Max: 100

**Panel: JVM Heap Usage per Service**
```promql
jvm_memory_used_bytes{area="heap", job=~"$service"}
/
jvm_memory_max_bytes{area="heap", job=~"$service"}
* 100
```
- Legend: `{{job}}` · Unit: Percent

**Panel: HikariCP Pending Connections**
```promql
hikaricp_connections_pending{job=~"$service"}
```
- Thresholds: 1=yellow, 5=red

### Dashboard 2: ClinicCare Application Performance

**Panel: Request Rate per Service (req/s)**
```promql
sum by (job) (rate(http_server_requests_seconds_count{job=~"$service"}[5m]))
```

**Panel: HTTP 5xx Error Rate (%)**
```promql
100 * sum by (job) (
  rate(http_server_requests_seconds_count{outcome="SERVER_ERROR", job=~"$service"}[5m])
)
/
sum by (job) (
  rate(http_server_requests_seconds_count{job=~"$service"}[5m])
)
```
- Thresholds: 1=yellow, 5=red · Unit: Percent

**Panel: Response Time — p50 / p95 / p99**
```promql
# p50
histogram_quantile(0.50, sum by (job, le) (rate(http_server_requests_seconds_bucket{job=~"$service"}[5m])))
# p95
histogram_quantile(0.95, sum by (job, le) (rate(http_server_requests_seconds_bucket{job=~"$service"}[5m])))
# p99
histogram_quantile(0.99, sum by (job, le) (rate(http_server_requests_seconds_bucket{job=~"$service"}[5m])))
```
- Unit: Seconds · Thresholds: 0.5=yellow, 1.0=red

**Panel: Appointments Booked (last hour)**
```promql
increase(appointments_booked_total{job="appointment-service"}[1h])
```
- Visualization: Stat · Colour: Green

**Panel: Appointment Booking p95 Duration**
```promql
histogram_quantile(0.95, sum by (le) (rate(appointment_booking_duration_seconds_bucket[5m])))
```
- Visualization: Gauge · Unit: Seconds · Min: 0, Max: 3 · Thresholds: 0.5=green, 1.0=yellow, 2.0=red

**Panel: Notification Success vs Failures (last hour)**
```promql
# Query A (sent)
increase(notifications_sent_total[1h])
# Query B (failed)
increase(notification_failures_total[1h])
```
- Visualization: Bar Gauge · Legend: `{{channel}} - {{__name__}}`

**Panel: Active Doctor Count**
```promql
active_doctors_online
```
- Visualization: Stat · Colour mode: From thresholds (0=red, 1=green)

## 4.5 Dashboard Best Practices

1. **Name and document everything** — every panel needs a title and description. A year from now someone needs to understand it immediately.
2. **Consistent time ranges** — use `$__rate_interval` for rate functions; never hardcode windows.
3. **Use recording rules in dashboards** — if a query takes > 1 second, it belongs in a recording rule. Target < 100ms per query.
4. **Group related panels with rows** — Overview → HTTP Performance → Business Metrics → Infrastructure.
5. **Include deployment annotations** — immediately shows "latency spiked after this deployment."
6. **Avoid over-crowding** — 3 focused dashboards with 10-15 panels each beats one dashboard with 50 panels.
7. **Test in darkness** — review dashboards off-hours. Do they make sense when you are half-asleep and something is on fire? If not, simplify.
8. **Use links between dashboards** — from the overview, panels should link to the detail dashboard for that service (use "Data links" in panel settings).

## 4.6 Grafana Alerting (Unified Alerting)

Grafana Unified Alerting (introduced in 8.x) allows defining alerts directly in Grafana.

| | Prometheus Alerting | Grafana Unified Alerting |
|---|---|---|
| **Strength** | Close to data; fast evaluation | Multi-datasource (Prometheus + Loki together) |
| **UI** | Prometheus Alerts tab | Richer alert management UI |
| **Recommended for** | Infrastructure + metric-only alerts | Cross-datasource, complex conditions |

**Creating a Grafana Alert (overview):**
1. Open a panel → **Edit → Alert tab**
2. Define the condition (e.g., `avg() of query A > 0.05`)
3. Set evaluation interval and `for` duration
4. Configure notification channels (contact points)
5. Save

Grafana contact points support: Email, Slack, PagerDuty, OpsGenie, Teams, Webhook, and more.

## 4.7 Kubernetes Monitoring Architecture

Kubernetes introduces new monitoring dimensions: nodes, pods, deployments, services, namespaces, persistent volumes.

### Key Components

**kube-state-metrics** — watches the Kubernetes API and exposes object-level metrics:
```
kube_pod_status_phase{namespace, pod, phase}
kube_deployment_status_replicas_available{namespace, deployment}
kube_node_status_condition{node, condition, status}
kube_pod_container_resource_limits{container, resource}
```
Answers: *"How many pods are running? Are deployments healthy?"*

**cAdvisor (Container Advisor)** — built into the kubelet, exposes per-container resource metrics:
```
container_cpu_usage_seconds_total{container, namespace, pod}
container_memory_working_set_bytes{container, namespace, pod}
container_network_receive_bytes_total{pod, namespace}
container_fs_reads_bytes_total{container}
```
Answers: *"How much CPU/memory is each pod consuming?"*

**Node Exporter (DaemonSet)** — same Node Exporter from Day 1, deployed as a DaemonSet so it runs on every node.

**Metrics Server (optional)** — lightweight aggregator for `kubectl top`. Powers HPA. Does **not** replace Prometheus.

### Kubernetes Metrics Flow

```
[kubelet / cAdvisor]          → Prometheus scrapes → [Prometheus]
[kube-state-metrics]          → Prometheus scrapes → [Prometheus]
[Node Exporter DaemonSet]     → Prometheus scrapes → [Prometheus]
[ClinicCare pods /actuator/prometheus] → Prometheus scrapes → [Prometheus]
                                                          ↓
                                                       [Grafana]
```

## 4.8 Deploying kube-prometheus-stack via Helm

`kube-prometheus-stack` installs in one Helm command:
- Prometheus Operator
- Prometheus
- Alertmanager
- Grafana
- kube-state-metrics
- Node Exporter (DaemonSet)
- Pre-built dashboards and alerts for Kubernetes

### Step 1 — Add Helm Repo

```bash
helm repo add prometheus-community https://prometheus-community.github.io/helm-charts
helm repo update
```

### Step 2 — Create Values File (`cliniccare-monitoring-values.yml`)

```yaml
grafana:
  enabled: true
  adminPassword: cliniccare123
  service:
    type: NodePort
    nodePort: 30300

prometheus:
  prometheusSpec:
    retention: 15d
    storageSpec:
      volumeClaimTemplate:
        spec:
          storageClassName: standard
          resources:
            requests:
              storage: 10Gi
    additionalScrapeConfigs:
      - job_name: 'cliniccare-services'
        kubernetes_sd_configs:
          - role: pod
            namespaces:
              names: ['cliniccare']
        relabel_configs:
          - source_labels: [__meta_kubernetes_pod_annotation_prometheus_io_scrape]
            action: keep
            regex: true
          - source_labels: [__meta_kubernetes_pod_annotation_prometheus_io_path]
            action: replace
            target_label: __metrics_path__
            regex: (.+)
          - source_labels: [__address__, __meta_kubernetes_pod_annotation_prometheus_io_port]
            action: replace
            regex: ([^:]+)(?::\d+)?;(\d+)
            replacement: $1:$2
            target_label: __address__
          - action: labelmap
            regex: __meta_kubernetes_pod_label_(.+)
          - source_labels: [__meta_kubernetes_namespace]
            target_label: kubernetes_namespace
          - source_labels: [__meta_kubernetes_pod_name]
            target_label: kubernetes_pod_name

alertmanager:
  enabled: true
```

### Step 3 — Install

```bash
kubectl create namespace monitoring

helm install kube-prometheus \
  prometheus-community/kube-prometheus-stack \
  --namespace monitoring \
  --values cliniccare-monitoring-values.yml
```

### Step 4 — Verify

```bash
kubectl get pods -n monitoring      # All pods should be Running
kubectl get svc -n monitoring       # Find Grafana NodePort
```

### Step 5 — Access Grafana

```bash
kubectl port-forward svc/kube-prometheus-grafana 3000:80 -n monitoring
# Open http://localhost:3000 — admin / cliniccare123
```

### Step 6 — Explore Pre-Built Dashboards

- Kubernetes / Compute Resources / Node
- Kubernetes / Compute Resources / Namespace (Pods)
- Kubernetes / Compute Resources / Pod
- Node Exporter / Full
- Prometheus / Overview

## 4.9 Deploying ClinicCare on Kubernetes

Each pod must have Prometheus scrape annotations:

```yaml
# Example — appointment-service Deployment
apiVersion: apps/v1
kind: Deployment
metadata:
  name: appointment-service
  namespace: cliniccare
spec:
  replicas: 2
  selector:
    matchLabels:
      app: appointment-service
  template:
    metadata:
      labels:
        app: appointment-service
      annotations:
        prometheus.io/scrape: "true"              # Enable scraping
        prometheus.io/port:   "8082"              # Port to scrape
        prometheus.io/path:   "/actuator/prometheus"  # Metrics path
    spec:
      containers:
        - name: appointment-service
          image: cliniccare/appointment-service:1.0.0
          ports:
            - containerPort: 8082
          resources:
            requests: { memory: "256Mi", cpu: "100m" }
            limits:   { memory: "512Mi", cpu: "500m" }
          readinessProbe:
            httpGet: { path: /actuator/health/readiness, port: 8082 }
            initialDelaySeconds: 30
            periodSeconds: 10
          livenessProbe:
            httpGet: { path: /actuator/health/liveness, port: 8082 }
            initialDelaySeconds: 60
            periodSeconds: 30
```

```bash
# Deploy all services
kubectl create namespace cliniccare
kubectl apply -f k8s/ -n cliniccare

# Verify
kubectl get pods -n cliniccare
```

In **Prometheus UI → Status → Targets** — all five services should appear with state **UP** and label `kubernetes_namespace="cliniccare"`.
