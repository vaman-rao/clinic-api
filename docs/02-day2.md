## 2.1 Advanced PromQL Functions

### Rate Functions (for Counters)

```promql
# rate() — per-second average over the window. Best for dashboards & trends.
rate(appointments_booked_total{job="appointment-service"}[5m])

# irate() — instant rate using only the last two data points. Best for alerting on spikes.
irate(http_server_requests_seconds_count{uri="/api/appointments"}[5m])

# increase() — total increase over the window. Best for "how many X happened?"
increase(billing_invoices_generated_total[24h])
```

> `increase()` is syntactic sugar for `rate() * duration_seconds`. It may return non-integer values due to extrapolation — this is normal.

### Aggregation Operators

| Operator | Description |
|---|---|
| `sum()` | Sum all matching series |
| `avg()` | Average all matching series |
| `min()` | Minimum value across series |
| `max()` | Maximum value across series |
| `count()` | Count of time-series (not values) |
| `topk(n, metric)` | Top N series by value |
| `bottomk(n, metric)` | Bottom N series by value |
| `quantile(φ, metric)` | φth quantile (0 ≤ φ ≤ 1) |

**Using `by()` and `without()` clauses:**

```promql
# Total request rate per service
sum by (job) (rate(http_server_requests_seconds_count[5m]))

# Error rate per endpoint (keep uri, drop everything else)
sum by (uri) (rate(http_server_requests_seconds_count{status=~"5.."}[5m]))

# Average JVM memory usage, ignoring instance (multi-instance services)
avg without (instance) (jvm_memory_used_bytes{area="heap"})
```

**ClinicCare example — requests per service:**
```promql
sum by (job) (rate(http_server_requests_seconds_count[5m]))
```
Result:
```
{job="appointment-service"}  23.4
{job="patient-service"}      11.2
{job="billing-service"}       4.7
{job="doctor-service"}        8.1
{job="notification-service"}  2.3
```

### Histogram Functions

```promql
# 95th percentile response time for appointment-service
histogram_quantile(
  0.95,
  sum by (le) (
    rate(http_server_requests_seconds_bucket{job="appointment-service"}[5m])
  )
)

# 99th percentile grouped by service
histogram_quantile(
  0.99,
  sum by (job, le) (
    rate(http_server_requests_seconds_bucket[5m])
  )
)
```

> Always `sum by (le)` before `histogram_quantile()`. When multiple instances exist, you must aggregate bucket counts across instances first (keeping the `le` label), then compute the quantile on the aggregated distribution.

### Temporal & Predictive Functions

```promql
# predict_linear — "will this disk fill up in 4 hours?"
predict_linear(node_filesystem_avail_bytes{mountpoint="/"}[1h], 4 * 3600) < 0

# delta — change in a gauge over a window (gauges only)
delta(appointment_queue_depth[10m])

# changes — how many times did a value change?
changes(active_doctors_online[1h])

# absent — fires if metric has NO data (service down / exporter failed)
absent(up{job="appointment-service"})
```

### Offset Modifier

```promql
# Is the current error rate higher than yesterday's at the same time?
rate(http_server_requests_seconds_count{status=~"5.."}[5m])
/
rate(http_server_requests_seconds_count{status=~"5.."}[5m] offset 24h)
```

## 2.2 Aggregations & Vector Matching

When you use arithmetic or comparison operators between two instant vectors, Prometheus must match labels between the two sides.

### One-to-one matching (default)

```promql
# Error rate as a fraction of total requests (per service)
sum by (job) (rate(http_server_requests_seconds_count{status=~"5.."}[5m]))
/
sum by (job) (rate(http_server_requests_seconds_count[5m]))
```

Both sides must have the **same label set** for matching to work.

### Ignoring / On clauses

```promql
# Ignore a label during matching
metric_a ignoring (pod) / metric_b

# Match only on specified labels
metric_a on (job) / metric_b
```

### Practical ClinicCare Example — HTTP Error % per Service

```promql
100 * sum by (job) (
    rate(http_server_requests_seconds_count{outcome="SERVER_ERROR"}[5m])
  )
/
  sum by (job) (
    rate(http_server_requests_seconds_count[5m])
  )
```

Result:
```
{job="appointment-service"}  0.23
{job="billing-service"}      1.87   ← elevated errors!
{job="patient-service"}      0.00
```

## 2.3 Recording Rules

Recording rules pre-compute expensive PromQL expressions and store the result as a new time-series. This makes dashboards and alerts significantly faster.

**When to use recording rules:**
- Queries used in multiple dashboards
- Expensive computations (`histogram_quantile` over many services)
- Queries that take > 1 second to evaluate

### Syntax

```yaml
# cliniccare_recording_rules.yml
groups:
  - name: cliniccare_http
    interval: 30s           # Evaluate every 30s (optional override)
    rules:
      - record: job:http_requests:rate5m     # Convention: level:metric:operation
        expr: >
          sum by (job) (
            rate(http_server_requests_seconds_count[5m])
          )

      - record: job:http_errors:rate5m
        expr: >
          sum by (job) (
            rate(http_server_requests_seconds_count{outcome="SERVER_ERROR"}[5m])
          )

      - record: job:http_error_ratio:rate5m
        expr: >
          job:http_errors:rate5m / job:http_requests:rate5m

      - record: job:http_request_duration_p99:rate5m
        expr: >
          histogram_quantile(0.99,
            sum by (job, le) (
              rate(http_server_requests_seconds_bucket[5m])
            )
          )
```

### Naming Convention

```
level:metric_name:aggregation_operation

Examples:
  job:appointments_booked:rate5m
  job:billing_revenue:increase1h
  cluster:http_requests:rate5m
```

### Referencing in `prometheus.yml`

```yaml
rule_files:
  - /etc/prometheus/rules/cliniccare_recording_rules.yml
  - /etc/prometheus/rules/cliniccare_alert_rules.yml
```

Reload without restart:
```bash
curl -X POST http://localhost:9090/-/reload
```

Verify: **Prometheus UI → Status → Rules** — all rules should show `OK`.

## 2.4 Service Discovery Concepts

Static configuration (`static_configs`) works for fixed environments. In Kubernetes and cloud environments, services come and go dynamically. Service discovery automatically finds new targets.

### Discovery Mechanisms

| Config Key | Discovers |
|---|---|
| `kubernetes_sd_configs` | Pods, services, nodes in Kubernetes |
| `consul_sd_configs` | Services registered in Consul |
| `ec2_sd_configs` | AWS EC2 instances |
| `docker_sd_configs` | Docker containers |
| `file_sd_configs` | Target lists from JSON/YAML files |

### Kubernetes Service Discovery — Preview

Prometheus queries the Kubernetes API for pods with specific annotations:

```yaml
# Annotations on a Kubernetes pod
annotations:
  prometheus.io/scrape: "true"
  prometheus.io/port:   "8082"
  prometheus.io/path:   "/actuator/prometheus"
```

Prometheus discovers this pod automatically and begins scraping it.

### Relabeling

Service discovery generates lots of metadata labels (pod name, namespace, node). Relabeling lets you:
- Keep only useful labels
- Rename labels
- Drop targets you don't want to monitor
- Create synthetic labels from discovery metadata

*(Explored in detail in the Day 4 Kubernetes lab.)*

## 2.5 Exporters Ecosystem

Exporters bridge Prometheus to systems that don't natively expose `/metrics`.

| Exporter | Purpose |
|---|---|
| **Node Exporter** | Linux/macOS host metrics |
| **Windows Exporter** | Windows host metrics |
| **Blackbox Exporter** | Probes HTTP/TCP/ICMP endpoints for availability |
| **JMX Exporter** | Java JVM metrics (for non-Spring apps) |
| **MySQL Exporter** | MySQL / MariaDB database metrics |
| **PostgreSQL Exporter** | PostgreSQL metrics |
| **Redis Exporter** | Redis metrics |
| **Kafka Exporter** | Apache Kafka consumer group metrics |
| **Elasticsearch Exporter** | Elasticsearch cluster / index metrics |

> For ClinicCare, the services use Spring Boot + Micrometer. No separate JVM exporter is needed — Micrometer handles it natively.

## 2.6 Monitoring Java with Spring Boot Actuator

Spring Boot Actuator exposes operational information about a running application. **Micrometer** acts as a vendor-neutral metrics façade — with a single dependency, you can export to Prometheus, Datadog, New Relic, CloudWatch, and many other backends.

### Dependencies (`pom.xml`)

```xml
<dependencies>
  <!-- Spring Boot Actuator -->
  <dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-actuator</artifactId>
  </dependency>

  <!-- Micrometer Prometheus Registry -->
  <dependency>
    <groupId>io.micrometer</groupId>
    <artifactId>micrometer-registry-prometheus</artifactId>
    <scope>runtime</scope>
  </dependency>
</dependencies>
```

### Actuator Configuration (`application.yml`)

```yaml
management:
  endpoints:
    web:
      exposure:
        include: health, info, prometheus, metrics
      base-path: /actuator
  endpoint:
    health:
      show-details: always
    prometheus:
      enabled: true
  metrics:
    tags:
      application: ${spring.application.name}  # Adds app label to ALL metrics
      environment: development

server:
  port: 8082  # appointment-service port

spring:
  application:
    name: appointment-service
```

Metrics exposed at: `http://localhost:8082/actuator/prometheus`


## 2.7 Micrometer Integration — Custom Business Metrics

### Counter — Tracking appointment bookings

```java
@Service
public class AppointmentService {

    private final Counter bookingsCounter;
    private final Counter cancellationsCounter;

    public AppointmentService(MeterRegistry registry) {
        this.bookingsCounter = Counter.builder("appointments.booked.total")
            .description("Total appointments successfully booked")
            .tag("type", "outpatient")
            .register(registry);

        this.cancellationsCounter = Counter.builder("appointments.cancelled.total")
            .description("Total appointments cancelled")
            .register(registry);
    }

    public Appointment bookAppointment(AppointmentRequest request) {
        Appointment apt = appointmentRepository.save(new Appointment(request));
        bookingsCounter.increment();
        return apt;
    }

    public void cancelAppointment(Long id) {
        appointmentRepository.deleteById(id);
        cancellationsCounter.increment();
    }
}
```

### Gauge — Tracking queue depth

```java
@Service
public class AppointmentQueueService {

    private final AtomicInteger queueDepth = new AtomicInteger(0);

    public AppointmentQueueService(MeterRegistry registry) {
        Gauge.builder("appointment.queue.depth", queueDepth, AtomicInteger::get)
            .description("Appointments waiting to be confirmed")
            .register(registry);
    }

    public void addToQueue(Appointment apt) {
        queue.add(apt);
        queueDepth.incrementAndGet();
    }

    public void processFromQueue() {
        Appointment apt = queue.poll();
        if (apt != null) queueDepth.decrementAndGet();
    }
}
```

### Timer (Histogram) — Measuring booking duration

```java
@Service
public class AppointmentService {

    private final Timer bookingTimer;

    public AppointmentService(MeterRegistry registry) {
        this.bookingTimer = Timer.builder("appointment.booking.duration")
            .description("Time to complete appointment booking end-to-end")
            .publishPercentileHistogram()     // enables histogram_quantile in PromQL
            .sla(                             // SLA bucket boundaries
                Duration.ofMillis(100),
                Duration.ofMillis(500),
                Duration.ofSeconds(1),
                Duration.ofSeconds(2)
            )
            .register(registry);
    }

    public Appointment bookAppointment(AppointmentRequest request) {
        return bookingTimer.record(() -> {
            validatePatient(request.getPatientId());
            checkDoctorAvailability(request.getDoctorId(), request.getSlot());
            Appointment apt = appointmentRepository.save(new Appointment(request));
            notificationService.sendConfirmation(apt);
            return apt;
        });
    }
}
```

### DistributionSummary — Tracking invoice amounts

```java
@Service
public class BillingService {

    private final DistributionSummary invoiceAmountSummary;
    private final Counter invoicesGenerated;

    public BillingService(MeterRegistry registry) {
        this.invoiceAmountSummary = DistributionSummary
            .builder("billing.invoice.amount")
            .description("Distribution of invoice amounts in INR")
            .baseUnit("INR")
            .publishPercentileHistogram()
            .register(registry);

        this.invoicesGenerated = Counter.builder("billing.invoices.generated.total")
            .description("Total invoices generated")
            .register(registry);
    }

    public Invoice generateInvoice(Long appointmentId) {
        Invoice invoice = buildInvoice(appointmentId);
        invoiceAmountSummary.record(invoice.getAmount());
        invoicesGenerated.increment();
        return invoice;
    }
}
```

**Resulting PromQL queries:**
```promql
# 95th percentile invoice amount
histogram_quantile(0.95, rate(billing_invoice_amount_bucket[5m]))

# Average invoice amount
rate(billing_invoice_amount_sum[5m]) / rate(billing_invoice_amount_count[5m])
```
