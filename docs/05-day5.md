## 5.1 End-to-End Monitoring Flow

Let's trace the complete journey of a metric from a ClinicCare service all the way to an alert landing in a Slack channel.

```
Step 1 — INSTRUMENTATION (Code)
  Developer adds Timer to AppointmentService.bookAppointment()
  Micrometer creates: appointment_booking_duration_seconds_bucket/_sum/_count

Step 2 — EXPOSITION (HTTP /metrics)
  Spring Boot Actuator exposes all metrics at:
  http://appointment-service:8082/actuator/prometheus

Step 3 — SCRAPING (Prometheus pulls)
  Every 15s: HTTP GET /actuator/prometheus
  Adds labels: job="appointment-service", instance="appointment-service:8082"
  Stores (timestamp, value) pairs in local TSDB

Step 4 — RULE EVALUATION
  Every 15s: histogram_quantile(0.99, ...) > 3.0
  If true → alert goes PENDING
  If true for 5m → alert goes FIRING

Step 5 — ALERT DISPATCH
  Prometheus: POST http://alertmanager:9093/api/v2/alerts

Step 6 — ROUTING & NOTIFICATION
  Alertmanager groups the alert, checks inhibition, looks up routing tree
  severity=critical → receiver=slack-backend
  After group_wait (30s) → sends to Slack webhook

Step 7 — RESOLUTION
  Booking latency drops below 3.0s
  Prometheus: FIRING → INACTIVE
  Alertmanager sends resolution message to Slack

Step 8 — VISUALISATION (Grafana)
  Engineers view "ClinicCare Application Performance" dashboard to:
  - Confirm p99 latency was elevated
  - Check when it started (annotation)
  - Correlate with error rate and request volume
  - Verify recovery after fix
```

**Prometheus Exposition Format** (what `/actuator/prometheus` returns):
```
# HELP appointment_booking_duration_seconds Time to complete booking
# TYPE appointment_booking_duration_seconds histogram
appointment_booking_duration_seconds_bucket{le="0.1"} 1024
appointment_booking_duration_seconds_bucket{le="0.5"} 2048
appointment_booking_duration_seconds_bucket{le="1.0"} 2100
appointment_booking_duration_seconds_bucket{le="+Inf"} 2103
appointment_booking_duration_seconds_sum   412.3
appointment_booking_duration_seconds_count 2103
```

## 5.2 Prometheus Federation

Federation allows one Prometheus server to scrape metrics from another. This enables hierarchical or multi-cluster monitoring architectures.

### Use Cases

**a) Cross-Datacenter Aggregation:**  
Multiple Prometheus instances (one per data centre) each scrape their local ClinicCare services. A global Prometheus scrapes aggregated metrics from all regional Prometheuses.

**b) Long-Term Storage Separation:**  
Short-term Prometheus (2 weeks) scrapes all metrics. A federation Prometheus scrapes only important aggregated recording rules and stores them for 1+ year.

### Federation Configuration

```yaml
# In the "global" Prometheus prometheus.yml
scrape_configs:
  - job_name: 'federate-region-ap-south'
    honor_labels: true              # Preserve original labels from source
    metrics_path: '/federate'       # Prometheus federation endpoint
    params:
      match[]:                      # Which metrics to federate
        - '{job=~".*-service"}'     # All ClinicCare services
        - 'node_cpu_seconds_total'
        - 'node_memory_MemAvailable_bytes'
    static_configs:
      - targets: ['prometheus-ap-south:9090']

  - job_name: 'federate-region-ap-north'
    honor_labels: true
    metrics_path: '/federate'
    params:
      match[]:
        - '{job=~".*-service"}'
    static_configs:
      - targets: ['prometheus-ap-north:9090']
```

### Important Rules for Federation

- Only federate what you need (use tight `match[]` selectors)
- Federation `scrape_interval` should be ≥ source `scrape_interval`
- High-cardinality metrics are expensive to federate
- Use recording rules on the source Prometheus to pre-aggregate before federating

## 5.3 Remote Write

Remote write allows Prometheus to continuously stream metric samples to a remote storage system. This is the **modern alternative** to federation for long-term storage.

### Architecture

```
[Prometheus] --remote_write--> [Thanos Receive / Cortex / Mimir / VictoriaMetrics]
                                          ↓
                           [Long-term Object Storage: S3 / GCS / Azure Blob]
                                          ↓
                           [Query via Thanos Query / Grafana]
```

### Configuration

```yaml
# In prometheus.yml
remote_write:
  - url: "http://thanos-receive:10908/api/v1/receive"
    queue_config:
      max_samples_per_send: 10000
      max_shards:           200
      capacity:             2500
    write_relabel_configs:
      # Only remote-write metrics from ClinicCare services
      - source_labels: [job]
        regex:  '.*-service'
        action: keep
```

### When to Use Remote Write vs Federation

| | Remote Write | Federation |
|---|---|---|
| **Best for** | Long-term storage, multi-tenant, cloud-native | Simple multi-datacenter aggregation |
| **Infrastructure** | Requires Thanos/Mimir/Cortex | Just another Prometheus instance |
| **Recommended** | ✅ Modern default | Simple setups only |

## 5.4 Introduction to Thanos

Thanos extends Prometheus with:
1. **Unlimited retention** via object storage (S3, GCS, Azure Blob)
2. **Global query view** across multiple Prometheus instances
3. **High availability (HA)** with deduplication
4. **Downsampling** for fast long-term queries

### Thanos Components

| Component | Role |
|---|---|
| **Thanos Sidecar** | Runs alongside each Prometheus pod. Uploads TSDB blocks to object storage. Exposes gRPC API for Thanos Query. |
| **Thanos Store** | Reads data from object storage. Makes historical data queryable without loading into memory. |
| **Thanos Query** | Global query frontend. Fans out PromQL queries to all sidecars and stores; deduplicates results. |
| **Thanos Compactor** | Applies downsampling (5m and 1h resolutions) to old data; enforces retention policies. |
| **Thanos Receive** | Alternative to sidecar. Accepts `remote_write` data directly. |

### ClinicCare Production Pattern with Thanos

```
dev-prometheus  ──┐
staging-prometheus ──┤── Thanos Sidecar ──► Object Storage (S3)
prod-prometheus ──┘                              ↑
                                           Thanos Store
                                                 ↑
                                          Thanos Query  ◄── Grafana
```

- Each environment (dev, staging, prod) has its own Prometheus
- Thanos Query provides a unified view across all environments
- Grafana connects to Thanos Query instead of individual Prometheuses
- Historical data (> 2 weeks) stored in S3 at reduced cost

## 5.5 Securing Prometheus & Grafana (RBAC Basics)

In development, Prometheus and Grafana run without authentication. **In production, always secure them.**

### Prometheus Security

**TLS (Prometheus 2.24+):**
```yaml
# In prometheus.yml
tls_server_config:
  cert_file: /etc/prometheus/tls/prometheus.crt
  key_file:  /etc/prometheus/tls/prometheus.key
```

**Basic Authentication:**
```yaml
basic_auth_users:
  admin: $2y$12$...   # bcrypt hash
# Generate: htpasswd -nBC 12 "" | tr -d ':\n'
```

**In Kubernetes:** expose Prometheus only within the cluster. Access via `kubectl port-forward` or through a service mesh with mTLS. Never expose Prometheus NodePort to the internet.

### Grafana Security

**Change default credentials immediately:**
```bash
GF_SECURITY_ADMIN_PASSWORD=<strong-password>
```

**RBAC — Built-in Roles:**

| Role | Permissions |
|---|---|
| **Viewer** | View dashboards. Cannot edit. |
| **Editor** | Create and edit dashboards. |
| **Admin** | Full access including user management. |

**ClinicCare recommended assignments:**
- Engineering team → Editor
- Operations → Editor
- Management/stakeholders → Viewer
- On-call engineer → Admin (during incidents)

**Teams and Folders:**
```
billing-team:  Viewer of all folders, Editor of "Billing" folder
backend-team:  Editor of "ClinicCare Application" folder
platform-team: Admin of "Infrastructure" folder
```

**OAuth / SSO Integration:**
```bash
GF_AUTH_GOOGLE_ENABLED=true
GF_AUTH_GOOGLE_CLIENT_ID=...
GF_AUTH_GOOGLE_CLIENT_SECRET=...
```
In production, use SSO (Google, GitHub, GitLab, Okta, LDAP) rather than local accounts.

### Kubernetes Network Security

```yaml
# NetworkPolicy — only Grafana and Alertmanager can reach Prometheus
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-prometheus-access
  namespace: monitoring
spec:
  podSelector:
    matchLabels:
      app: prometheus
  ingress:
    - from:
        - podSelector: { matchLabels: { app: grafana } }
        - podSelector: { matchLabels: { app: alertmanager } }
      ports:
        - port: 9090
```

## 5.6 Monitoring Best Practices in Production

### Capacity Planning

```
bytes_per_sample ≈ 1.3 bytes (compressed)
samples/sec = num_targets × num_metrics_per_target × (1 / scrape_interval)

ClinicCare example:
  5 services  × 200 metrics × (1/15s) ≈  67 samples/sec
  Node Exporter: 800 metrics × (1/15s) ≈  53 samples/sec
  Total                                  ≈ 120 samples/sec

Storage per day = 120 × 86,400 × 1.3 bytes ≈ 13 MB/day
30-day retention                            ≈ 400 MB

Add 50% headroom for new metrics and cardinality growth.
```

### Retention Policy

```bash
# Docker run flags
--storage.tsdb.retention.time=30d
--storage.tsdb.retention.size=20GB   # Whichever comes first
```

### High Availability (HA)

Run two identical Prometheus instances scraping the same targets. Both store all data independently. Alertmanager deduplicates alerts from both. Grafana connects to both via a load balancer — or use Thanos for proper HA.

### Cardinality Management

High cardinality = too many unique label combinations = memory exhaustion.

**Detection query:**
```promql
topk(10, count by (__name__) ({__name__=~".+"}))
```
→ Shows the 10 metrics with the most time-series.

**Prevention:**
- Never use `user_id`, `request_id`, `session_id` as labels
- Audit new metrics before deploying to production
- Use built-in cardinality tools: `http://localhost:9090/api/v1/status/tsdb`

### Runbooks

Every alert must have a runbook answering:
1. What does this alert mean?
2. What is the user impact?
3. What are the first 3 things to check?
4. What are the possible causes?
5. How do I fix each cause?
6. How do I verify it's fixed?
7. Who else do I need to notify?

**ClinicCare Runbook — HighErrorRateCritical:**

| Field | Detail |
|---|---|
| **What** | `appointment-service` is returning > 5% HTTP 5xx responses |
| **Impact** | Patients cannot book appointments; UI shows error |
| **Check first** | `kubectl logs -n cliniccare deploy/appointment-service --tail=50` · Check HikariCP pending connections |
| **Possible causes** | Database down · DB connection pool exhausted · Bad deployment · Downstream service timeout |
| **Fix a** | Restart DB pod; check DB logs |
| **Fix b** | Increase connection pool; scale replicas |
| **Fix c** | `kubectl rollout undo deploy/appointment-service` |
| **Fix d** | Check `notification-service` health |
| **Verify** | Error rate in Grafana drops below 1% |
| **Notify** | Backend team lead, operations manager |

### Task 3 — Deploy Recording Rules and Alert Rules (30 min)

```yaml
# cliniccare-recording-rules.yml
apiVersion: v1
kind: ConfigMap
metadata:
  name: cliniccare-recording-rules
  namespace: monitoring
  labels:
    release: kube-prometheus    # Required for Operator to pick up
data:
  cliniccare-recording.yml: |
    groups:
      - name: cliniccare_recording
        interval: 30s
        rules:
          - record: job:http_requests_total:rate5m
            expr: >
              sum by (job) (rate(http_server_requests_seconds_count[5m]))
          - record: job:http_error_ratio:rate5m
            expr: >
              sum by (job) (
                rate(http_server_requests_seconds_count{outcome="SERVER_ERROR"}[5m])
              )
              /
              sum by (job) (
                rate(http_server_requests_seconds_count[5m])
              )
```

```yaml
# cliniccare-prometheusrule.yml
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: cliniccare-alerts
  namespace: monitoring
  labels:
    release: kube-prometheus
spec:
  groups:
    - name: cliniccare_availability
      rules:
        - alert: ClinicCareServiceDown
          expr: up{job=~".*-service"} == 0
          for: 1m
          labels:
            severity: critical
          annotations:
            summary: "Service {{ $labels.job }} is down"

        - alert: HighErrorRateCritical
          expr: job:http_error_ratio:rate5m > 0.05
          for: 5m
          labels:
            severity: critical
          annotations:
            summary: "Error rate > 5% on {{ $labels.job }}"
```

```bash
kubectl apply -f cliniccare-recording-rules.yml -n monitoring
kubectl apply -f cliniccare-prometheusrule.yml -n monitoring
```

Verify: **Prometheus UI → Status → Rules** → `cliniccare_availability` group visible, rules in `INACTIVE` state.

### Task 5 — Simulate Failure Scenarios (30 min)

**Scenario A — Service Crash (Critical Alert)**
```bash
# Delete appointment-service pod to simulate a crash
kubectl delete pod -n cliniccare -l app=appointment-service

# Kubernetes restarts it, but there's a ~30-60s outage window
# Watch in Prometheus → Alerts: ClinicCareServiceDown PENDING → FIRING
# Check Alertmanager UI: http://localhost:9093
# Once pod restarts, confirm alert resolves
```

**Scenario B — High Error Rate (Gradual Degradation)**
```bash
# Scale to 0 replicas
kubectl scale deploy appointment-service --replicas=0 -n cliniccare

# Generate traffic
while true; do
  curl -s http://localhost:8082/api/appointments > /dev/null
  sleep 0.5
done

# Watch error rate climb in Grafana
# After 5 minutes: HighErrorRateCritical fires

# Restore
kubectl scale deploy appointment-service --replicas=2 -n cliniccare
# Observe recovery — alert should resolve
```

**Scenario C — Memory Pressure**
```bash
# If appointment-service has a memory leak simulation endpoint:
curl -X POST http://localhost:8082/test/leak?mb=200

# Watch JVM Heap panel in Grafana climb
# Observe JVMHeapCritical alert if heap > 90%
# Restart pod to resolve
kubectl rollout restart deploy/appointment-service -n cliniccare
```
